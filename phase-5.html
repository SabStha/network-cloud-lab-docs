Phase 05 Branch Office

How to build this — AWS lab

The branch isn't a peered or shared network — it's a completely separate, independent VPC, built the same way HQ was in Phase 1. That's the whole point: two real sites only ever meet over the internet through the tunnel, never through a backdoor AWS connection.

Why not just peer the two VPCs?AWS VPC Peering would connect them directly, bypassing the whole reason for a VPN. Two genuinely separate VPCs, joined only by an encrypted tunnel over their public IPs, is what actually proves the site-to-site design works — exactly like two real offices, each with their own ISP.
  • A second VPC — not connected to HQ's in any way yet.
    Show me how
    1. AWS Console → VPC → Your VPCs → Create VPC
    2. Resources to create: VPC only
    3. Name tag: branch-vpc
    4. IPv4 CIDR: 10.20.0.0/16
    5. Click Create VPC
  • Same pattern as HQ: one room facing the street, one that doesn't.
    Show me how
    1. VPC Console → Subnets → Create subnet
    2. VPC ID: select branch-vpc
    3. Subnet 1 — Name branch-public, IPv4 CIDR 10.20.1.0/24
    4. Add new subnet — Name branch-private, IPv4 CIDR 10.20.20.0/24
    5. Click Create subnet
  • Show me how
    1. VPC Console → Internet Gateways → Create internet gateway
    2. Name tag: branch-igw → Create
    3. Select it → Actions → Attach to VPC → choose branch-vpc
  • Show me how
    1. VPC Console → Route Tables → Create route table
    2. Public: Name branch-public-rt, VPC branch-vpc → add route 0.0.0.0/0 → Internet Gateway branch-igw → associate with branch-public
    3. Private: Name branch-private-rt, VPC branch-vpc → associate with branch-private, leave routes for later
  • A second, fully independent firewall — HQ's config has no idea this one exists yet.
    Show me how
    1. AWS Marketplace → Palo Alto Networks VM-Series → Bundle 1 (PAYG) → Launch through EC2
    2. Name: branch-palo-alto · Instance type: same as HQ (e.g. c5n.xlarge)
    3. Key pair: create a new one, e.g. branch-palo-alto
    4. Network settings: VPC branch-vpc, Subnet branch-public, Auto-assign public IP: Enable
    5. Security group: new, allow HTTPS(443)/SSH(22) from My IP
    6. Launch instance — allow 15–45 minutes to fully boot, same as HQ
  • Show me how
    1. EC2 → Network Interfaces → Create network interface → Subnet branch-public, Description branch-untrust-eth1, SG allow-all → Create
    2. Create another → Subnet branch-private, Description branch-trust-eth2, SG allow-all → Create
    3. Attach both to the branch-palo-alto instance via Actions → Attach to instance
  • Show me how
    1. EC2 → Elastic IPs → Allocate Elastic IP address
    2. Associate it with branch-untrust-eth1
    3. Write this address down — you'll need it for the HQ side of the tunnel in Phase 6
  • Show me how
    1. EC2 → Network Interfaces → select branch-trust-eth2
    2. Actions → Change source/destination check → Stopped → Save
  • Show me how
    1. EC2 → branch-palo-alto → Security tab → management SG → Inbound rules → add HTTPS(443) and SSH(22) from My IP
    2. Untrust/trust SGs: allow all traffic from 0.0.0.0/0 — same reasoning as HQ
What's not done yetBoth firewalls now exist independently, each fully able to reach the internet on its own — but they don't know about each other at all. The tunnel interface addresses (10.255.0.1 / 10.255.0.2) and the actual IKE/IPSEC configuration that connects them live entirely inside PAN-OS, not AWS — that's all of Phase 6.
