Enterprise Network → Hybrid Cloud Infrastructure

Built the way a real infrastructure team builds it: requirements first, design on paper before a single interface is touched, then implementation in eleven sequential phases — from a single firewall to a Terraform-managed AWS footprint.

Client: Example Corp Platform: Palo Alto PAN-OS · AWS · Terraform Format: 11 phases, each with Objective → Design → IP Plan → Config → Validation → Docs

Kickoff

Business Case & Requirements

Every design decision in this document traces back to a stated requirement below — nothing is configured just because it's possible.

The Company

Example Corp — a 50-employee company with one headquarters office, a set of internal servers, and existing internet access. No formal network segmentation or firewall policy exists yet. A branch office is planned, and a future migration to AWS is on the roadmap.

Requirements

  • Secure, filtered internet access for all staff
  • Network segmentation between departments and functions
  • Stateful firewall protection at the perimeter
  • Encrypted site-to-site connection to a future branch
  • A design that migrates cleanly into a cloud provider
  • Centralized monitoring and traffic visibility

Glossary — jargon used in this documentclick to expand
CIDR (/24, /30…)
Shorthand for subnet size — the number after the slash is how many bits are the network portion; fewer bits free for hosts means a smaller subnet.
VLAN
A virtual LAN — splits one physical switch into multiple isolated broadcast domains.
Trunk
A switch port that carries traffic for multiple VLANs at once, tagged by VLAN ID.
Zone (Palo Alto)
A named group of interfaces/networks that security policy is written between, e.g. Trust → Untrust.
NAT
Network Address Translation — rewrites private IPs to a public IP for internet-bound traffic.
IKE / IPSEC
IKE negotiates a secure key exchange; IPSEC uses that key to encrypt the actual VPN traffic.
DH Group
Diffie-Hellman group — the mathematical strength of the key exchange; higher number, stronger key.
VPC
Virtual Private Cloud — an isolated, software-defined network inside AWS.
IAM
Identity and Access Management — controls who/what can do what inside AWS.
Policy Matrix
A source → destination → service → action table that defines every allowed firewall path explicitly.

Phase 2 of 10

IP Address Design

Addressing is planned on paper before it's typed into a device — every subnet size is a deliberate choice, not a default.

Design — WAN Link

The ISP hand-off is a point-to-point link between the ISP router and the firewall's WAN interface.

Why a /30?A point-to-point link only ever has two devices on it. A /30 gives exactly 2 usable addresses — no wasted space, and it signals to anyone reading the plan that this is a transit link, not a LAN.
DeviceInterfaceIP Address
ISP Router—203.0.113.1/30
Palo Altoethernet1/1 (WAN)203.0.113.2/30
Design — LAN Network

A single private block is reserved for the whole company, then broken into function-based subnets.

Why a /16 for 50 employees?50 people don't need 65,000 addresses today — but re-numbering a live network later is painful. A /16 (CIDR 10.10.0.0/16) leaves headroom to add sites, VLANs, and device classes for years without ever touching this decision again.

Phase 3 of 10

Firewall Design

Zones come before rules. Every rule that follows references one of these four named groups — nothing is written ad hoc.

Design — Security Zones
UNTRUST

Internet

Public, unauthenticated networks. Deny inbound by default.

TRUST

Employee Network

Corporate endpoints. Outbound allowed, inbound restricted.

SERVER

Internal Services

Application/data tier. Accepts only explicitly permitted services.

DMZ

Public Services

Internet-facing apps, isolated from Trust and Server.

Phase 4 of 10

Implementation & Testing

Only now — after requirements, topology, addressing, zones, and policy are all on paper — does the firewall get touched.

Configuration Tasks — build order

  1. Configure interfaces

    Assign ethernet1/1 (WAN) and ethernet1/2 (LAN) the addresses from the Phase 2 IP plan.

  2. Assign zones

    Bind each interface to its Phase 3 zone — WAN → Untrust, LAN → Trust.

  3. Configure the virtual router

    Create the routing instance that owns both interfaces.

  4. Configure the default route

    0.0.0.0/0 next hop 203.0.113.1 — sends all unmatched traffic to the ISP.

  5. Create security rules

    Enter the Phase 3 policy matrix into the rulebase, top to bottom, in order.

  6. Create NAT

    Source-NAT internal subnets to the WAN IP so private addresses can reach the internet.

Phase 6 of 10

IPSEC VPN Design

The cryptographic parameters for the tunnel introduced in Phase 5 — defined on paper before either firewall's VPN wizard is opened.

Design — IKE (Phase 1) Parameters

IKEv2

Key exchange version — faster rekeying than IKEv1, native MOBIKE support.

AES-256

Symmetric encryption for the negotiated key material.

SHA-256

Integrity hash (HMAC) for the exchange.

DH Group 14

2048-bit Diffie-Hellman group for Phase 1 key strength.

Design — IPSEC (Phase 2) Parameters
ParameterValue
EncryptionAES-256
AuthenticationSHA-256
Lifetime3600 seconds
Traffic selector10.10.0.0/16 ↔ 10.20.0.0/16
Why re-key every hour?A shorter IPSEC lifetime limits how much traffic is ever encrypted under one key — if a key were somehow compromised, the exposure window is capped at 3600 seconds, not the life of the tunnel.
