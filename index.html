Enterprise Network & Hybrid Cloud Infrastructure Lab

Reference architecture for a secure enterprise network built on a Palo Alto perimeter, switched VLAN segmentation, and site-to-site IPSEC — extended into an AWS hybrid cloud footprint and codified with Terraform.

Author: Network/Cloud Engineering Status: Design & Lab Build Platform: Palo Alto PAN-OS · AWS · Terraform
01

Project Overview

A ground-up design and lab build of an enterprise network, progressing from a single-site firewall foundation to a fully segmented, monitored, and hybrid-cloud-connected infrastructure managed as code.

Business Scenario

A growing company needs a secure, segmented network at its headquarters, a resilient encrypted link to a new branch office, and a path to extend workloads into AWS without re-architecting security policy. The design must support internal user, server, and guest traffic separation, centralized firewall policy enforcement, and auditable logging for compliance.

Objectives

  • Establish a hardened perimeter with Palo Alto NGFW
  • Segment traffic by function using VLANs and zones
  • Connect HQ and branch over site-to-site IPSEC VPN
  • Enforce least-privilege access with policy matrices
  • Centralize logging and monitoring for visibility
  • Extend the trust model into AWS via hybrid connectivity
  • Codify the environment with Terraform for repeatability

Technologies Used

Security

Project Evolution Roadmap

Phase 1

Palo Alto Firewall Foundation

Stand up the NGFW, configure interfaces, zones, and NAT, and validate baseline internet access for the internal segment.

Phase 2

Branch Office + Site-to-Site IPSEC VPN

Deploy a second firewall at the branch and establish an encrypted IKEv2/IPSEC tunnel back to HQ.

Phase 3

Switching + VLAN Segmentation

Introduce trunked switching and separate Admin, Users, and Servers into dedicated VLANs.

Phase 4

Security Zones + Firewall Policies

Map VLANs to firewall zones and enforce least-privilege policy between them.

Phase 5

Logging + Monitoring

Centralize traffic, threat, and session logs; establish baseline dashboards and alerting.

Phase 6

AWS Hybrid Cloud Migration

Extend the on-prem trust boundary into an AWS VPC over IPSEC, hosting EC2 and database workloads.

Phase 7

Terraform Infrastructure as Code

Codify AWS (and where possible, network) configuration for repeatable, version-controlled deployment.

04

Firewall Design

Zones map directly to VLAN segments; policy between them defaults to deny and is opened only for required, explicitly defined services.

UNTRUST

Internet

Public, unauthenticated networks. Default posture: deny inbound, App-ID controlled outbound.

TRUST

Users

Corporate endpoints (VLAN 20). Outbound internet allowed; access to Servers restricted by service.

SERVER

Internal Servers

Application and data tier (VLAN 30). Accepts only explicitly permitted services from Trust.

DMZ

Public Applications

Internet-facing services isolated from both Trust and Server zones.

05

VPN Design

A route-based IPSEC tunnel secures HQ-to-branch traffic using current, non-deprecated IKEv2 cryptographic standards.

IKEv2

Key exchange protocol; supports MOBIKE and faster rekeying than IKEv1.

AES-256

Symmetric encryption for the data-plane ESP payload.

SHA-256

Hash algorithm for integrity verification (HMAC).

DH Group 14

2048-bit Diffie-Hellman group for Phase 1 key material.

Cloud Migration Design

On-prem services are lifted into a segmented AWS VPC, replacing implicit network trust with explicit, auditable controls.

  • On-prem servers only
After

Future Improvements

Planned iterations that move the lab from a manually built environment toward a fully automated, observable infrastructure.

PAN-OS XML API scripting for policy validation and drift detection.

CI/CD

Pipeline-driven Terraform plan/apply with peer-reviewed changes.

Unified view of firewall, VPN, and cloud metrics via CloudWatch + Grafana.
