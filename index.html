Glossary — plain-English definitionstap to expand
CIDR (/24, /30…)
Shorthand for subnet size. Bigger number after the slash = fewer addresses in that subnet.
VLAN
Splits one switch into several separate mini-networks that can't see each other by default.
Trunk
A cable that carries several VLANs at once, each traffic frame tagged with its VLAN number.
Zone (Palo Alto)
A named group of networks the firewall writes rules between, e.g. "Trust" vs "Untrust."
NAT
Swaps a private IP for a public one so internal devices can reach the internet.
IKE / IPSEC
IKE agrees on a secret key between two firewalls; IPSEC uses that key to encrypt the actual traffic.
DH Group
How mathematically strong the key exchange is — higher number, harder to crack.
VPC
A private, walled-off network inside AWS, same idea as your office LAN.
IAM
Controls exactly who or what is allowed to do what inside AWS.
Policy Matrix
A table listing every path traffic is allowed to take — nothing gets through that isn't on the list.

v1 topology

How to build this — AWS lab

You don't have physical hardware, so this v1 topology gets built with EC2 instances inside a VPC instead. Every box in the diagram above maps to a real AWS resource.

Why build the VPC before launching any EC2 instances?They're not alternatives — EC2 is the computer, the VPC is the room it sits in. Every EC2 instance has to be launched into a subnet, and a subnet only exists inside a VPC, so there's no way to skip straight to "just launch EC2." AWS does give every account a flat, fully-public "default VPC" you could launch into instead — but that has no segmentation at all, which would defeat the point of this whole project. Build the network first, then move the firewall and PC-1 into it.
Where did the ISP router go?In AWS you don't deploy an "ISP router" yourself — an Internet Gateway (IGW) attached to your VPC plays that role. It isn't your home/office internet router either; that's outside this topology entirely, it's just how your laptop reaches the AWS Console. The IGW is virtual, free, and needed from the very start of Phase 1 — without it the VPC has no path to the internet at all, so nothing else in this phase can be tested.

Each task below shows a small picture of exactly what it changes, right above the click-by-click steps — the picture is the point, the text is backup.

  • One boundary. Nothing inside it yet.
    Show me how
    1. AWS Console → search VPC → Your VPCs → Create VPC
    2. Resources to create: VPC only (not the wizard — build it piece by piece)
    3. Name tag: enterprise-lab-vpc
    4. IPv4 CIDR: 10.10.0.0/16
    5. IPv6 CIDR block: No IPv6 CIDR block · Tenancy: Default
    6. Click Create VPC
  • Two rooms — one will face the street, one won't.
    Show me how
    1. VPC Console → Subnets → Create subnet
    2. VPC ID: select enterprise-lab-vpc
    3. Subnet 1 — Name public-wan, Availability Zone: any (e.g. us-east-1a), IPv4 CIDR 10.10.1.0/24
    4. Click Add new subnet
    5. Subnet 2 — Name private-lan, same or a different AZ, IPv4 CIDR 10.10.20.0/24
    6. Click Create subnet
  • The whole VPC now has a door to the internet — but it's not connected to a room yet.
    Show me how
    1. VPC Console → Internet Gateways → Create internet gateway
    2. Name tag: enterprise-lab-igw → Create internet gateway
    3. It shows as "Detached" — select it → Actions → Attach to VPC
    4. Choose enterprise-lab-vpc → Attach internet gateway
  • Public room now routes to the internet. Private room's route is still pending.
    Show me how
    1. VPC Console → Route Tables → Create route table
    2. Public: Name public-rt, VPC enterprise-lab-vpc → Create
    3. Select it → Routes tab → Edit routes → Add route: Destination 0.0.0.0/0, Target → Internet Gateway → enterprise-lab-igw → Save
    4. Subnet associations tab → Edit → check public-wan → Save
    5. Private: Name private-rt, same VPC → Create
    6. Subnet associations tab → Edit → check private-lan → Save
    ⚠ Leave its 0.0.0.0/0 route for now — it needs to point at the firewall's LAN network interface, which doesn't exist until Step 5. Come back after that.
  • The firewall lands in the public room — nothing else touches it yet.
    Show me how
    1. AWS Marketplace → search Palo Alto Networks VM-Series
    2. Pick VM-Series Next-Generation Firewall Bundle 1 (pay-as-you-go — no separate license needed for a lab)
    3. Continue to Subscribe → Accept Terms → Continue to Configuration → pick your region → Continue to Launch
    4. Choose Action: Launch through EC2 — this opens the normal EC2 launch wizard with the AMI already selected
    5. Name: hq-palo-alto · Instance type: m5.xlarge
    6. Key pair: create a new one (you'll want it for CLI access later)
    7. Network settings → Edit: VPC enterprise-lab-vpc, Subnet public-wan, Auto-assign public IP: Enable (temporary — replaced by a proper Elastic IP in a later step)
    8. Security group: create new, allow HTTPS(443) and SSH(22) from My IP only
    9. Click Launch instance
    ⚠ PAN-OS takes 10–15 minutes to fully boot the first time — much longer than a normal Linux instance. Don't worry if it's not reachable immediately.
  • Three doors: management, internet-facing, and one reaching into the private room.
    Show me how
    1. Launching in Step 5 already gave the firewall one interface — management, in public-wan. This step adds the other two.
    2. EC2 Console → Network Interfaces → Create network interface
    3. Subnet: public-wan · Description: untrust-eth1 · Security group: allow-all (PAN-OS does the real filtering, not this SG) → Create
    4. Create another → Subnet: private-lan · Description: trust-eth2 · Security group: allow-all → Create
    5. Select the untrust-eth1 interface → Actions → Attach to instance → choose hq-palo-alto → Attach
    6. Select the trust-eth2 interface → Actions → Attach to instance → choose hq-palo-alto → Attach
    7. EC2 → Instances → hq-palo-alto → Networking tab → confirm you now see 3 interfaces total
  • The internet-facing door now has a real, fixed public address.
    Show me how
    1. EC2 Console → Elastic IPs → Allocate Elastic IP address → Allocate
    2. Select the new address → Actions → Associate Elastic IP address
    3. Resource type: Network interface
    4. Network interface: choose untrust-eth1
    5. Private IP: its only option should auto-select → Associate
    ⚠ Consider allocating a second Elastic IP for the management interface too — the "auto-assign public IP" from Step 5 isn't fixed and can change if the instance ever stops/starts.
  • Now the firewall is allowed to forward traffic that isn't addressed to itself.
    Show me how
    1. EC2 Console → Network Interfaces → select trust-eth2
    2. Actions → Change source/destination check
    3. Set to Stopped (i.e. checking is disabled) → Save
    ⚠ Do this on the trust interface at minimum. Skipping it is the single most common reason a firewall NVA in AWS silently drops all forwarded traffic.
  • The gate to the admin UI only opens for you. Everything else, PAN-OS decides.
    Show me how
    1. EC2 → Instances → hq-palo-alto → Security tab → click the management security group
    2. Inbound rules → Edit inbound rules → Add rule
    3. Type: HTTPS · Source: My IP → this is the PAN-OS admin UI
    4. Add another rule: Type SSH · Source: My IP (optional, for CLI access) → Save rules
    5. Repeat for the untrust/trust security groups, but allow all traffic from 0.0.0.0/0 instead — PAN-OS itself decides what's actually allowed through; the AWS security group here is intentionally wide open
    ⚠ If you can't reach the PAN-OS login page, this is the step to check first — a missing inbound HTTPS rule is the most common cause, not routing.

Phase 06IPSEC VPN Design The lock and key that actually encrypts the branch tunnel. Not started
In one sentence: both firewalls agree on a shared secret (IKE), then use it to scramble every packet that crosses the tunnel (IPSEC).
IKE (Phase 1) parameters

IKEv2

Key exchange version — faster rekeying, native MOBIKE.

AES-256

Encryption for the negotiated key material.

SHA-256

Integrity hash for the exchange.

DH Group 14

2048-bit key-exchange strength.

IPSEC (Phase 2) parameters
ParameterValue
EncryptionAES-256
AuthenticationSHA-256
Lifetime3600 seconds
Traffic selector10.10.0.0/16 ↔ 10.20.0.0/16
Why re-key every hour?A shorter lifetime limits how much traffic is ever encrypted under one key — if a key leaked, the exposure window is capped at 3600 seconds.
