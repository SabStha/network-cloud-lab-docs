Overview/Phase 06
Phase 06 IPSEC VPN Design
The lock and key that actually encrypts the branch tunnel. Not started
In one sentence: both firewalls agree on a shared secret (IKE), then use it to scramble every packet that crosses the tunnel (IPSEC).
IKE (Phase 1) parameters

IKEv2

Key exchange version — faster rekeying, native MOBIKE.

AES-256

Encryption for the negotiated key material.

SHA-256

Integrity hash for the exchange.

DH Group 14

2048-bit key-exchange strength.

IPSEC (Phase 2) parameters
ParameterValue
EncryptionAES-256
AuthenticationSHA-256
Lifetime3600 seconds
Traffic selector10.10.0.0/16 ↔ 10.20.0.0/16
Why re-key every hour?A shorter lifetime limits how much traffic is ever encrypted under one key — if a key leaked, the exposure window is capped at 3600 seconds.
How to build this — in PAN-OS, not AWS

This entire phase happens inside the two firewalls' web UIs — nothing here is an AWS resource. Every step gets done twice: once on HQ's firewall, once on Branch's, with the addresses mirrored.

Two ends, one tunnelA tunnel needs both sides configured before it comes up. If you only configure HQ, it'll sit there trying to negotiate with a Branch that isn't listening. Do each step below on both firewalls before moving to the next step.
  • Show me how
    1. Log into PAN-OS → Network → Network Profiles → IKE Crypto → Add
    2. Name: ike-crypto-aes256
    3. Encryption: aes-256-cbc · Authentication: sha256 · DH Group: group14
    4. Key Lifetime: 8 hours (the IKE session itself, separate from the IPSec data lifetime)
    5. Repeat identically on the Branch firewall
  • Each gateway object points at the other firewall's public IP, with a matching pre-shared key.
    Show me how
    1. Network → IKE Gateways → Add
    2. Name: to-branch (on HQ) / to-hq (on Branch)
    3. Version: IKEv2 only mode
    4. Interface: ethernet1/1 (the untrust interface) · Local IP Address: this firewall's Elastic IP
    5. Peer IP Address Type: IP · Peer Address: the other firewall's Elastic IP
    6. Pre-shared Key: set the same secret string on both sides · Local/Peer Identification: leave as IP
    7. Advanced Options tab → IKE Crypto Profile: ike-crypto-aes256
    8. Repeat with addresses swapped on the other firewall
  • Show me how
    1. Network → Network Profiles → IPSec Crypto → Add
    2. Name: ipsec-crypto-aes256
    3. ESP Encryption: aes-256-cbc · Authentication: sha256
    4. Lifetime: 3600 seconds
    5. Repeat identically on the Branch firewall
  • Show me how
    1. Network → IPSec Tunnels → Add
    2. Name: to-branch-tunnel (HQ) / to-hq-tunnel (Branch)
    3. Tunnel Interface: click the dropdown → New Tunnel Interface → create tunnel.1
    4. IKE Gateway: select the gateway from step 2 · IPSec Crypto Profile: select the profile from step 3
    5. Proxy IDs tab → Add: Local 10.10.0.0/16, Remote 10.20.0.0/16 (HQ) — reversed on Branch
  • A tunnel interface needs its own small /30-style address, same as any other interface.
    Show me how
    1. Network → Interfaces → Tunnel → select tunnel.1
    2. Config tab: Virtual Router = same router as your other interfaces · Security Zone = trust (or a dedicated VPN zone)
    3. IPv4 tab: Add → 10.255.0.1/30 on HQ, 10.255.0.2/30 on Branch
    4. Network → Virtual Routers → your router → Static Routes → add 10.20.0.0/16 via tunnel.1 (HQ) / 10.10.0.0/16 via tunnel.1 (Branch)
  • Show me how
    1. Policies → Security → Add
    2. Source Zone: Trust · Destination Zone: Trust (or your VPN zone) · Action: Allow
    3. Do this on both firewalls — the tunnel carries the packets, the policy decides whether they're allowed through
